# Инструкция по работе с шаблонным репозиторием

Этот репозиторий предназначен для учебных заданий по Python. В нём уже настроены базовая структура проекта, проверки качества кода, тесты, CI и шаблон Pull Request. Ниже описано, как начать работу и как правильно вести разработку.

## 1) Как установить Python

В шаблоне используется **Python 3.13**.

Проверить установленную версию можно командой:

```bash
python --version
```

или:

```bash
python3 --version
```

Если Python не установлен, его нужно установить с официального сайта Python или через менеджер пакетов вашей операционной системы. После установки снова проверьте версию командой `python --version`.


## 2) Что делать после создания репозитория на GitHub

После получения репозитория от преподавателя нужно открыть терминал в папке проекта и выполнить следующие шаги.

### 2.1. Склонировать репозиторий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
```

### 2.2. Установить `uv`

`uv` — это инструмент для управления зависимостями, виртуальным окружением и запуском команд проекта.

Проверьте, установлен ли он:

```bash
uv --version
```

Если `uv` не установлен, но Python уже есть, его можно поставить так:

```bash
python -m pip install uv
```

### 2.3. Установить зависимости проекта

В корне репозитория выполните:

```bash
uv sync --group dev
```

Эта команда создаст окружение и установит зависимости из `pyproject.toml`, включая инструменты разработки.

### 2.4. Установить `pre-commit`

`pre-commit` запускает автоматические проверки перед коммитом.

Установка:

```bash
uv run pre-commit install
```

После этого при каждом `git commit` будут запускаться проверки форматирования и статического анализа.

Чтобы проверить все файлы вручную, используйте:

```bash
uv run pre-commit run --all-files
```

### 2.5. Быстрая проверка проекта

Перед отправкой кода полезно запускать:

```bash
uv run ruff check .
uv run ruff format --check .
uv run mypy .
uv run pytest
```

Если нужно автоматически исправить форматирование и часть замечаний Ruff:

```bash
uv run ruff check . --fix
uv run ruff format .
```

## 3) Как и где писать код

В этом шаблоне используется **плоская структура проекта**, без `src/` layout.

Пример структуры:

```text
template-repository/
├── pyproject.toml
├── .pre-commit-config.yaml
├── README.md
├── INSTRUCTIONS.md
├── <YOUR_APP_NAME_RENAME_ME>/
│   ├── __init__.py
│   └── main.py
└── tests/
    ├── __init__.py
    └── test_example.py
```

### Куда писать код

- Основной код проекта пишите в пакетах, которые будут лежать в `<YOUR_APP_NAME_RENAME_ME>`
- Тесты пишите в папке `tests/`.
- В `README.md` хранится описание проекта, зачем он нужен, инструкции к запуску и настройки.
- Техническая инструкция по работе с репозиторием и питон-таском лежит в `INSTRUCTIONS.md`.

### Как оформлять модули

- Для каждого отдельного блока логики старайтесь создавать отдельный файл.
- Не складывайте весь код в один большой файл.
- Тесты должны проверять поведение кода, а не только наличие строк.
- По возможности называйте файлы, классы и функции понятно и коротко.
- Корень репозитория не должен превращаться в склад случайных `.py` файлов. Лучше группировать код по пакетам.

## 4) Как пользоваться Git: кратко и по делу

Git нужен для того, чтобы несколько людей могли работать над проектом без потери истории изменений.

### Основные команды

Проверить состояние:

```bash
git status
```

Посмотреть, какие файлы изменились:

```bash
git diff
```

Добавить файл в коммит:

```bash
git add <файл>
```

Добавить всё изменённое:

```bash
git add .
```

Сделать коммит:

```bash
git commit -m "Короткое и понятное сообщение"
```

Отправить изменения на сервер:

```bash
git push
```

Получить обновления из удалённого репозитория:

```bash
git pull
```

Посмотреть ветки:

```bash
git branch
```

Переключиться на другую ветку:

```bash
git checkout <имя-ветки>
```

Создать новую ветку и перейти в неё:

```bash
git checkout -b <имя-ветки>
```

### Как работать в ветке

Правильный процесс такой:

1. Получить репозиторий.
2. Создать отдельную ветку для задачи.
3. Внести изменения в код и тесты.
4. Проверить проект локально.
5. Сделать коммит.
6. Отправить ветку в удалённый репозиторий.
7. Создать Pull Request в `main`.

Пример:

```bash
git checkout -b feature/add-solution
```

Потом вы меняете код, проверяете его, сохраняете изменения:

```bash
git add .
git commit -m "Добавил решение задачи"
git push origin feature/add-solution
```

После этого на GitHub создаёте Pull Request в ветку `main`.

### Как сделать Pull Request

Когда ветка отправлена на GitHub:

- откройте репозиторий на GitHub;
- нажмите кнопку создания Pull Request;
- выберите ветку с вашими изменениями и цель `main`;
- заполните шаблон PR;
- отправьте на проверку.

после клонирования переименуйте your_app_name в имя вашего проекта;
используйте имя пакета в snake_case;
основной код храните внутри этого пакета;
не складывайте .py файлы россыпью в корень.

В описании PR нужно писать не менее 100 символов. Это помогает преподавателю или ревьюеру быстро понять, что сделано и зачем.

## 5) Что делать нельзя и почему

### Нельзя пушить напрямую в ветку `main`

Работа напрямую в `main` ломает нормальный процесс разработки. В команде это плохо, потому что:

- трудно понять, кто и что изменил;
- легко случайно сломать рабочую версию проекта;
- невозможно нормально проверить код через PR;
- история изменений становится беспорядочной.

### Нельзя игнорировать тесты и проверки

Если код не проходит линтеры, проверку типов и тесты, он может выглядеть рабочим только на первый взгляд. Ошибки лучше поймать до отправки кода, чем после.

### Нельзя коммитить временные и лишние файлы

Не нужно добавлять в репозиторий:

- виртуальные окружения;
- кэш тестов;
- сгенерированные отчёты;
- временные файлы редактора;
- `__pycache__`.

Такие файлы не относятся к исходному коду и только засоряют историю проекта.

## 6) Какие инструменты качества кода используются

В проекте используются несколько инструментов качества кода.

### Ruff

`Ruff` используется сразу в двух ролях:

- как линтер;
- как форматтер.

Он проверяет стиль, импорты, часть типичных ошибок Python и может автоматически исправлять часть замечаний.

### mypy

`mypy` проверяет типы. Он помогает находить ошибки в аннотациях и замечать проблемы ещё до запуска программы.

### pytest

`pytest` запускает тесты. С его помощью проверяется, что код работает так, как ожидается.

### pre-commit

`pre-commit` запускает автоматические проверки перед коммитом. В этом шаблоне он проверяет служебные файлы, запускает Ruff и mypy.

### Зачем это нужно вместе

- `Ruff` приводит код к единому стилю и ловит часть ошибок.
- `mypy` проверяет типы.
- `pytest` проверяет поведение программы.
- `pre-commit` автоматически запускает проверки перед коммитом.

## 7) Ограничения и правила, которые накладывают `pyproject.toml` и `.pre-commit-config.yaml`

Ниже перечислены ключевые ограничения, которые теперь действуют в этом шаблоне.

### Версия Python

- Код должен быть совместим с **Python 3.13**.
- Нельзя ориентироваться на старые версии Python, если для решения есть современный синтаксис.

### Форматирование и стиль

- Форматирование выполняется через `ruff format`.
- `Black` в этом шаблоне не используется.
- Строки форматируются в стиле с **двойными кавычками**.
- Импорты автоматически приводятся к единому порядку.

### Аннотации типов

- Аннотации обязательны **для всех функций**, включая тесты.
- Требуются аннотации аргументов и возвращаемого значения.
- Частично аннотированные функции тоже считаются ошибкой.
- Проверка типов запускается не только на аннотированном коде, но и на телах функций.

### Современный typing-стиль

В шаблоне запрещён устаревший стиль из `typing`, если для него есть современная замена.

Нужно писать так:

```python
names: list[str]
config: dict[str, int]
value: str | None
result: int | str
```

Нельзя писать так:

```python
from typing import Dict, List, Optional, Union

names: List[str]
config: Dict[str, int]
value: Optional[str]
result: Union[int, str]
```

### Что проверяет Ruff

В шаблоне включены проверки на:

- базовые ошибки стиля и синтаксиса;
- неиспользуемые переменные и импорты;
- порядок импортов;
- потенциально опасные конструкции;
- именование;
- упрощение кода;
- обязательность аннотаций;
- современный синтаксис Python.

### Что разрешено

- `print` использовать **можно**. Для учебных задач это допустимо.
- Докстринги **не обязательны**.
- `self` и `cls` не нужно отдельно аннотировать.

### Как работает pre-commit

Перед коммитом автоматически запускаются:

- проверки служебных файлов, например YAML и TOML;
- исправление лишних пробелов и отсутствия перевода строки в конце файла;
- `ruff check --fix`;
- `ruff format`;
- `mypy`.

Если один из этих шагов падает, коммит нужно исправить и запустить снова.

## Короткий рабочий сценарий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
uv sync --group dev
uv run pre-commit install

git checkout -b feature/my-task
# пишете код в пакетах верхнего уровня и тесты в tests/
uv run pre-commit run --all-files
uv run pytest

git add .
git commit -m "Реализация задачи"
git push origin feature/my-task
```

После этого создайте Pull Request в `main` и заполните шаблон описания.
