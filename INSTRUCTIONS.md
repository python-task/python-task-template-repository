# Инструкция по работе с шаблонным репозиторием

Этот репозиторий предназначен для учебных заданий по Python.

В шаблоне уже настроены:

- базовая структура проекта;
- проверки качества кода;
- тесты;
- CI;
- шаблон Pull Request;
- базовые ограничения на работу с основной веткой.

Ниже описано, как начать работу, где писать код, как запускать проект, как сдавать задачу и какие правила важно соблюдать.

## 1) Базовые требования

### Версия Python

В шаблоне используется **Python 3.13**.

Проверьте установленную версию:

```bash
python --version
```

или:

```bash
python3 --version
```

Если Python не установлен, установите его с официального сайта Python или через менеджер пакетов вашей операционной системы.

### Обязательный инструмент для проекта

В этом шаблоне используется `uv`.

Он нужен для:

- создания и использования окружения;
- установки зависимостей;
- запуска проектных команд.

Проверьте, установлен ли он:

```bash
uv --version
```

Если `uv` не установлен, его можно поставить так:

```bash
python -m pip install uv
```

### Что важно знать про `uv`

- `uv sync --group dev` обычно **сам создаёт** виртуальное окружение `.venv` в корне проекта.
- В этом шаблоне **не нужно** сначала вручную делать `python -m venv .venv`.
- `uv` умеет использовать или подбирать подходящую версию Python для проекта.
- Если на компьютере нет подходящей версии Python, `uv` обычно может **сам скачать** её.

Но есть важный нюанс:

- это сработает только если у `uv` не отключена автозагрузка Python;
- и если у вас есть доступ в интернет.

Если автозагрузка Python отключена или сеть недоступна, тогда Python `3.13` нужно установить вручную.

## 2) Что делать после получения репозитория

После получения репозитория от преподавателя откройте терминал и выполните следующие шаги.

### 2.1. Склонировать репозиторий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
```

### 2.2. Установить зависимости

```bash
uv sync --group dev
```

Эта команда:

- создаст окружение `.venv`, если его ещё нет;
- подберёт подходящий Python для проекта;
- установит зависимости из `pyproject.toml`, включая инструменты разработки.

### 2.3. Установить `pre-commit`

```bash
uv run pre-commit install
```

После этого перед каждым `git commit` будут автоматически запускаться проверки.

### 2.4. Проверить, что всё работает

Полезный минимальный набор локальных проверок:

```bash
uv run ruff check .
uv run ruff format --check .
uv run mypy .
uv run pytest
```

Если нужно автоматически исправить часть замечаний:

```bash
uv run ruff check . --fix
uv run ruff format .
```

## 3) Как устроен этот шаблон

В шаблоне используется **flat layout**, без `src/`.

Пример структуры:

```text
your-repository/
├── pyproject.toml
├── .pre-commit-config.yaml
├── README.md
├── INSTRUCTIONS.md
├── YOUR_APP_NAME_RENAME_ME/
│   ├── __init__.py
│   ├── __main__.py
│   └── main.py
└── tests/
    ├── __init__.py
    └── test_example.py
```

## 4) Где писать код

### Основной код

После клонирования репозитория нужно:

1. Переименовать директорию `YOUR_APP_NAME_RENAME_ME` в имя своего проекта.
2. Использовать имя пакета в `snake_case`.
3. Хранить основной код внутри этого пакета.

Пример:

```text
my_amazing_solution/
├── __init__.py
├── __main__.py
├── main.py
└── service.py
```

### Тесты

Тесты нужно писать в директории `tests/`.

Примеры:

```text
tests/test_main.py
tests/test_service.py
```

### Что важно по структуре

- Не складывайте `.py` файлы россыпью в корень репозитория.
- Не держите всё решение в одном огромном файле.
- Разбивайте код на логические модули.
- Давайте файлам, функциям, классам и переменным понятные имена.

## 5) Как запускать проект

В шаблоне уже есть файл `__main__.py`. Это значит, что после переименования пакета проект удобно запускать как модуль.

Допустим, вы переименовали директорию `YOUR_APP_NAME_RENAME_ME` в `my_amazing_solution`.

Тогда запуск будет таким:

```bash
uv run python -m my_amazing_solution
```

Именно такой способ запуска предпочтителен для этого шаблона.

Если вы храните основную логику запуска в `__main__.py`, то:

- команда остаётся короткой;
- импортный контекст будет корректным;
- структура проекта остаётся аккуратной.

## 6) Что нужно знать про стиль и проверки

Правила ниже идут не по желанию, а из настроек шаблона.

### Python и typing

Код должен быть совместим с **Python 3.13**.

Используйте современный typing-стиль:

```python
names: list[str]
config: dict[str, int]
value: str | None
result: int | str
```

Не используйте устаревший стиль там, где есть современная замена:

```python
from typing import Dict, List, Optional, Union

names: List[str]
config: Dict[str, int]
value: Optional[str]
result: Union[int, str]
```

### Аннотации типов

Аннотации обязательны:

- для всех функций;
- для методов;
- для тестов;
- для аргументов;
- для возвращаемых значений.

Частично типизированные функции тоже считаются проблемой.

`self` и `cls` отдельно аннотировать не нужно.

### Форматирование и стиль

В шаблоне используются:

- `ruff check`;
- `ruff format`;
- `mypy`;
- `pytest`;
- `pre-commit`.

Что важно:

- используется `ruff format`, а не `black`;
- строки форматируются с **двойными кавычками**;
- импорты приводятся к единому порядку;
- длина строки в шаблоне ориентирована на **100 символов**;
- докстринги не обязательны;
- `print` для учебных задач использовать можно.

### Что реально проверяется автоматически

Автоматические проверки ловят, в том числе:

- ошибки стиля;
- ошибки импортов;
- неиспользуемые переменные и импорты;
- проблемы с именованием;
- часть потенциально опасных конструкций;
- отсутствие аннотаций;
- несовременный typing-стиль;
- ошибки типизации;
- падение тестов.

## 7) Что делает `pre-commit`

Перед коммитом автоматически запускаются:

- проверка служебных файлов;
- исправление лишних пробелов;
- исправление отсутствия перевода строки в конце файла;
- `ruff check --fix`;
- `ruff format`;
- `mypy`.

Если какой-то шаг упал, сначала исправьте проблему, потом повторите коммит.

## 8) Как работать с Git

### Базовый процесс

Правильный процесс работы такой:

1. Получить репозиторий.
2. Создать отдельную ветку от `main`.
3. Внести изменения в код и тесты.
4. Прогнать локальные проверки.
5. Сделать коммит.
6. Отправить ветку в удалённый репозиторий.
7. Создать Pull Request в `main`.

Пример:

```bash
git checkout -b feature/add-solution
```

После изменений:

```bash
git add .
git commit -m "Добавил решение задачи"
git push origin feature/add-solution
```

## 9) Что нельзя делать

### Нельзя работать напрямую в `main`

Для текущего шаблона ориентируйтесь на правило:

- не коммитьте в `main`;
- не ведите разработку в `main`;
- все изменения вносите в отдельной ветке;
- сдача идёт через Pull Request.

Даже если GitHub где-то позволит технически сделать лишнее действие, ваш нормальный процесс работы всё равно должен быть таким: **ветка -> PR -> проверки -> review**.

### Нельзя игнорировать проверки

Нельзя отправлять код, который:

- не проходит линтер;
- не форматируется;
- не проходит типизацию;
- не проходит тесты.

### Нельзя коммитить мусор

Не добавляйте в репозиторий:

- `.venv`;
- `__pycache__`;
- кэши тестов;
- временные файлы редактора;
- случайные отчёты и артефакты.

## 10) Как сдавать задачу через Pull Request

После того как ветка отправлена на GitHub:

1. Откройте репозиторий.
2. Создайте Pull Request в `main`.
3. Заполните шаблон PR.
4. Убедитесь, что описание PR написано содержательно.

Хорошая практика для описания PR:

- кратко написать, что сделано;
- пояснить, зачем это сделано;
- указать, как проверялось решение;
- при наличии связанной задачи сослаться на неё.

Ориентир по объёму описания PR — **около 100 символов и больше**, но главное не длина сама по себе, а содержательность.

## 11) Какие проверки и ограничения обычно действуют при merge

По умолчанию в текущем учебном процессе и в стандартной policy шаблона ориентируйтесь на такие правила:

- merge идёт через Pull Request;
- в основной ветке ожидается зелёный CI;
- базовые обязательные проверки обычно `lint` и `test`;
- обычно нужен минимум **один approve**;
- после нового push в PR старые approve могут устареть, поэтому после доработок ревью может понадобиться заново;
- незакрытые обсуждения в PR могут блокировать merge;
- review может требоваться именно от преподавателя или назначенного ревьюера.

Это значит, что перед запросом ревью стоит:

1. привести код в порядок локально;
2. запустить проверки;
3. только потом отправлять PR на проверку.

## 12) Как сдача связана с Telegram-ботом

В учебной инфраструктуре используется Telegram-бот.

Студентский сценарий обычно такой:

1. Пройти регистрацию в боте через `/start`.
2. Быть назначенным в группу.
3. Получить доступ к своему учебному репозиторию.
4. Сделать изменения в отдельной ветке.
5. Открыть PR.
6. Запросить ревью через бота.

Полезные student-команды:

- `/student_repos` — показать ваши репозитории;
- `/request_review` — запросить ручное ревью по открытому PR;
- `/cancel_review` — отменить pending-запрос ревью;
- `/llm_review` — запустить LLM-review по открытому PR;
- `/balance` — посмотреть баланс LLM-кредитов;
- `/profile` — посмотреть профиль;
- `/notifications` — настроить уведомления.

Важно:

- для ревью и LLM-review нужен **открытый PR**;
- если вы ещё не назначены в группу, основные student-команды могут быть недоступны.

## 13) Практический чек-лист перед отправкой PR

Перед отправкой задачи проверьте:

- код лежит внутри вашего пакета верхнего уровня;
- пакет переименован из `YOUR_APP_NAME_RENAME_ME`;
- тесты лежат в `tests/`;
- функции и тесты типизированы;
- используются современные аннотации;
- код не лежит россыпью в корне;
- `uv run pre-commit run --all-files` проходит;
- `uv run pytest` проходит;
- проект запускается через `uv run python -m <имя_вашего_пакета>`;
- Pull Request создан в `main`;
- описание PR заполнено нормально и по делу.

## 14) Короткий рабочий сценарий

```bash
git clone <url-репозитория>
cd <имя-репозитория>

uv sync --group dev
uv run pre-commit install

git checkout -b feature/my-task

# переименуйте YOUR_APP_NAME_RENAME_ME в имя своего пакета
# пишите код внутри своего пакета верхнего уровня
# пишите тесты в tests/

uv run python -m my_amazing_solution
uv run pre-commit run --all-files
uv run pytest

git add .
git commit -m "Реализация задачи"
git push origin feature/my-task
```

После этого:

1. создайте Pull Request в `main`;
2. заполните шаблон PR;
3. запросите ревью через бота, если это требуется вашим процессом.

ориентируйтесь в таком порядке:

1. явные требования преподавателя к конкретной задаче;

3. эта инструкция.
