# Инструкция по работе с шаблонным репозиторием

Этот репозиторий предназначен для учебных заданий по Python. В нём уже настроены базовая структура проекта, линтеры, форматтер, тесты, CI и шаблон Pull Request. Ниже описано, как начать работу и как правильно вести разработку.

## 1) Как установить Python

Рекомендуется использовать версию python не ниже 3.11

Проверить установленную версию можно командой:

```bash
python --version
```

или:

```bash
python3 --version
```

Если Python не установлен, его нужно установить с официального сайта python.org или через менеджер пакетов вашей операционной системы. После установки снова проверьте версию командой `python --version`.

Если в проекте используется файл `.python-version`, значит для локальной работы желательно поставить именно эту версию Python.

## 2) Что делать после создание репозитория в github?

После получения репозитория от преподавателя нужно открыть терминал в папке проекта и выполнить следующие шаги.

### 2.1. Склонировать репозиторий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
```

### 2.2. Установить `uv`

`uv` — это быстрый инструмент для установки и запуска Python-зависимостей. Он используется в этом шаблоне для установки пакетов и запуска команд проекта.

Установка `uv` зависит от вашей системы. Если он уже есть, проверьте это командой:

```bash
uv --version
```

Если `uv` не установлен, но уже установлен python, uv можно поставить через него:

```bash
python -m pip install uv
```


### 2.3. Установить зависимости проекта

В корне репозитория выполните:

```bash
uv sync --all-extras
```

Эта команда создаст окружение и установит зависимости из `pyproject.toml`.

### 2.4. Установить `pre-commit`

`pre-commit` запускает проверки автоматически перед коммитом.

Установка:

```bash
uv run pre-commit install
```

После этого при каждом `git commit` будут запускаться проверки форматирования и линтинга.

Чтобы проверить все файлы вручную, используйте:

```bash
uv run pre-commit run --all-files
```

### 2.5. Быстрая проверка проекта

Перед отправкой кода полезно запускать:

```bash
uv run ruff check .
uv run black --check .
uv run mypy .
uv run pytest
```

## 3) Как и где писать код

В этом шаблоне используется стандартная структура проекта:

```text
template-repository-new/
├── pyproject.toml
├── README.md
├── src/
│   └── <имя_пакета>/
│       └── __init__.py
└── tests/
    └── test_example.py
```

### Куда писать код

* Основной код проекта пишите в папке `src/<имя_пакета>/`.
* Тесты пишите в папке `tests/`.
* В корневом `README.md` хранится краткое описание проекта.
* Подробная инструкция по работе лежит в отдельном файле `INSTRUCTIONS.md`.

### Почему используется структура `src/`

Такой подход помогает избежать проблем с импортами и делает структуру проекта более правильной и понятной. Код не должен лежать просто в корне репозитория.

### Как оформлять модули

* Для каждого отдельного блока логики старайтесь создавать отдельный файл.
* Не складывайте весь код в один большой файл.
* Тесты должны проверять поведение кода, а не только наличие строк.
* По возможности называйте файлы и функции понятно и коротко.

## 4) Как пользоваться Git: кратко и по делу

Git нужен для того, чтобы несколько людей могли работать над проектом без потери истории изменений.

### Основные команды

Проверить состояние:

```bash
git status
```

Посмотреть, какие файлы изменились:

```bash
git diff
```

Добавить файл в коммит:

```bash
git add <файл>
```

Добавить всё изменённое:

```bash
git add .
```

Сделать коммит:

```bash
git commit -m "Короткое и понятное сообщение"
```

Отправить изменения на сервер:

```bash
git push
```

Получить обновления из удалённого репозитория:

```bash
git pull
```

Посмотреть ветки:

```bash
git branch
```

Переключиться на другую ветку:

```bash
git checkout <имя-ветки>
```

Создать новую ветку и перейти в неё:

```bash
git checkout -b <имя-ветки>
```

### Как работать в ветке

Правильный процесс такой:

1. Получить репозиторий.
2. Создать отдельную ветку для задачи.
3. Внести изменения в код и тесты.
4. Проверить проект локально.
5. Сделать коммит.
6. Отправить ветку в удалённый репозиторий.
7. Создать Merge Request или Pull Request в `main`.

Пример:

```bash
git checkout -b feature/add-solution
```

Потом вы меняете код, проверяете его, сохраняете изменения:

```bash
git add .
git commit -m "Добавил решение задачи"
git push origin feature/add-solution
```

После этого на GitHub создаёте Pull Request в ветку `main`.

### Как сделать Merge Request / Pull Request

Когда ветка отправлена на GitHub:

* откройте репозиторий на GitHub;
* нажмите кнопку создания Pull Request;
* выберите ветку с вашими изменениями и цель `main`;
* заполните шаблон PR;
* отправьте на проверку.

В описании PR нужно писать не менее 100 символов. Это помогает преподавателю или ревьюеру быстро понять, что сделано и зачем.

## 5) Что делать нельзя и почему

### Нельзя пушить напрямую в ветку `main`

Работа напрямой в `main` ломает нормальный процесс разработки. В команде это плохо, потому что:

* трудно понять, кто и что изменил;
* легко случайно сломать рабочую версию проекта;
* невозможно нормально проверить код через PR;
* история изменений становится беспорядочной.

В компаниях обычно работают через отдельные ветки и ревью кода. Это безопаснее и удобнее.

### Нельзя игнорировать тесты и проверки

Если код не проходит линтеры и тесты, он может выглядеть рабочим только на первый взгляд. Ошибки лучше поймать до отправки кода, чем после.

### Нельзя коммитить временные и лишние файлы

Не нужно добавлять в репозиторий:

* виртуальные окружения;
* кэш тестов;
* сгенерированные отчёты;
* временные файлы редактора;
* `__pycache__`.

Такие файлы не относятся к исходному коду и только засоряют историю проекта.

## 6) Какие линтеры и форматтеры используются

В проекте используются несколько инструментов качества кода.

### Ruff

`Ruff` проверяет стиль и ошибки в коде. Он ловит:

* ошибки импорта;
* неиспользуемые переменные;
* проблемы со стилем;
* некоторые типовые ошибки в Python-коде.

### Black

`Black` автоматически форматирует код. Его задача — привести код к единому стилю, чтобы у всех участников проекта был одинаковый формат.

### mypy

`mypy` проверяет типы. Он помогает находить ошибки в аннотациях и замечать проблемы ещё до запуска программы.

### pytest

`pytest` запускает тесты. С его помощью проверяется, что код работает так, как ожидается.

### Зачем это нужно вместе

* `Black` делает код единообразным.
* `Ruff` ловит ошибки и нарушения стиля.
* `mypy` проверяет типы.
* `pytest` проверяет поведение программы.

Перед отправкой задания рекомендуется запускать все проверки локально, чтобы не получать ошибки в CI.

## Короткий рабочий сценарий

```bash
git clone <url-репозитория>
cd <имя-репозитория>
uv sync --all-extras
uv run pre-commit install

git checkout -b feature/my-task
# пишете код в src/ и тесты в tests/
uv run pre-commit run --all-files
uv run pytest

git add .
git commit -m "Реализация задачи"
git push origin feature/my-task
```

После этого создайте Merge Request / Pull Request в `main` и заполните шаблон описания.
